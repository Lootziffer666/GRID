# Repo file manager via Git Data API

This change adds a complete file-management feature to the Painkiller Android app: users can browse a GitHub repository's tree, queue file operations (move, rename, delete, create folder, upload, ZIP extract), and commit all queued changes atomically via the Git Data API. The implementation spans a domain-layer orchestrator with path-safe tree compilation, an app-layer repository and ViewModel, and a Compose UI with German labels. The domain layer is well-tested with 27 tests covering the compiler and orchestrator.

Watch for: stale tree after commit means the user sees outdated file listings until manual reload (confirmed); no path validation on user-entered folder/rename names in the ViewModel before they reach the compiler (confirmed); the `currentLevelEntries` filter relies on GitHub returning tree-type entries for directories in recursive mode, which breaks for truncated trees (likely); unused `Base64` import in the ViewModel (confirmed).

## High-level view

The domain layer introduces a clean pipeline: `PendingChange` model describes intent, `TreeChangeCompiler` compiles intent into tree deltas with path-safety validation via `PathValidation`, and `RepoTreeOrchestrator` executes the full getRef → getCommit → getTree → compile → createBlobs → createTree → createCommit → updateRef sequence. Error mapping mirrors the existing `MultiFileCommitOrchestrator` pattern. The `updateRef` call enforces `force=false` both at the orchestrator level and with a `require` guard in the Ktor implementation.

The ViewModel manages a `StateFlow<RepoTreeUiState>` with breadcrumb navigation derived from path segments. After a successful commit, pending changes are cleared but the tree is not reloaded, leaving the UI showing a stale listing that no longer reflects the committed state. User-entered paths (folder names, rename targets) are passed directly into `PendingChange` without validation — the compiler rejects unsafe paths at commit time, but the entire batch fails with a generic message rather than identifying the specific bad entry at queue time.

The `truncated` flag from the GitHub tree response is captured in `RepoTreeLoadResult.Success` but discarded by the ViewModel — there's no UI indication when the tree listing is incomplete.

<details>
<summary>Issues (6)</summary>

1. **Stale tree after commit** — After a successful `commitChanges()`, the ViewModel clears `pendingChanges` but does not call `loadTree()` to refresh `treeEntries`. The user sees the pre-commit file listing. Add a `loadTree()` call (or equivalent state reset prompting reload) in the success branch of `commitChanges()`.
2. **No path validation on user input in ViewModel** — `addNewFolder`, `renameFile`, and `moveFile` pass user-entered strings directly as `targetPath` on `PendingChange` without any sanitization. The `TreeChangeCompiler` will reject unsafe paths at commit time, but the user gets no immediate feedback when queueing a bad name. Validate early (e.g., call `PathValidation.isSafeRepoPath`) and surface an error in UI state.
3. **Truncated tree silently ignored** — `currentLevelEntries` filters by `entry.type == TYPE_TREE` for folders, but when the recursive tree response is truncated, directory entries may be omitted. The ViewModel discards the `truncated` flag from `RepoTreeLoadResult.Success`. Add a UI warning when truncated is true.
4. **Unused `Base64` import** — `RepoTreeViewModel` imports `android.util.Base64` but never uses it (the base64 content comes pre-encoded from `SafFileReader`). Remove it.
5. **`getTree` URL construction uses string concatenation instead of Ktor's parameter API** — `KtorGithubGitDataApi.getTree` builds the URL with `"?recursive=1"` appended as a raw string rather than using Ktor's `parameter("recursive", "1")`. Inconsistent with the rest of the codebase and fragile if the base URL ever includes existing query params.
6. **`CreateTreeResponse` reused as `getTree` return type** — The response shape happens to match, but semantically `getTree` returns a tree snapshot while `createTree` returns the result of a write operation. Sharing the type couples read and write API contracts.

## Stale UI after atomic commit

When `commitChanges()` succeeds, the ViewModel transitions to:

```kotlin
it.copy(
    isCommitting = false,
    commitResult = CommitResultInfo(...),
    pendingChanges = emptyList(),
)
```

The `treeEntries` field retains the pre-commit snapshot. Moves, renames, and deletes that were just committed are still visible at their old paths. Uploads appear absent. The user must manually re-press "Verzeichnisbaum laden" to see the updated state. An automatic `loadTree()` after success (or clearing `treeEntries` with a prompt to reload) would prevent confusion.

## Path validation deferred to commit time

The ViewModel's `addNewFolder`, `renameFile`, and `moveFile` methods construct a `PendingChange` from raw user input:

```kotlin
fun addNewFolder(folderName: String) {
    val fullPath = if (current.isEmpty()) folderName else "$current/$folderName"
    addPendingChange(PendingChange(type = CREATE_FOLDER, ..., targetPath = fullPath))
}
```

If the user enters `../etc` or a blank string, the change is queued without error. The compiler throws `IllegalArgumentException` later, which the orchestrator maps to `InvalidInput`. But the entire commit fails with a single error message that doesn't identify which queued change is at fault. With multiple queued changes, debugging becomes guesswork. Validating at queue time lets the UI surface the specific offending input immediately.

## Tree navigation and truncation

The `currentLevelEntries` computed property filters for root level with:

```kotlin
if (currentPath.isEmpty() && entry.path.contains('/')) return@filter false
```

This relies on the GitHub API returning `type: "tree"` entries for directories in the recursive response. For large trees (over 100k entries or when `truncated: true`), directory entries may be omitted, making subdirectories invisible. The `truncated` boolean flows through `RepoTreeLoadResult.Success` but the ViewModel's `loadTree()` success handler discards it:

```kotlin
is RepoTreeLoadResult.Success -> _state.update {
    it.copy(treeEntries = result.entries, isLoadingTree = false, currentPath = "")
}
```

No state field captures `truncated`, and no UI element warns the user.

## Blob creation gating for `.gitkeep`

The orchestrator creates blobs only when `entry.content.isNotEmpty()`. `TreeChangeCompiler` marks `CREATE_FOLDER` entries with `content = ""` (empty string for `.gitkeep`), so these skip blob creation and go to `createTree` with inline empty content. This is intentional — GitHub accepts inline empty content — but the gating condition is implicit. A comment or a more explicit sentinel (like a nullable field) would make the contract between compiler and orchestrator less fragile.

## Test coverage gaps

Not tested: the `RepoTreeViewModel` (no unit tests), the `RepoTreeRepository` (no unit tests), `currentLevelEntries` filtering logic, and the stale-tree-after-commit behavior. The navigation logic in `currentLevelEntries` is pure (derives from state fields alone) and could be extracted to a testable function in `:domain`.
