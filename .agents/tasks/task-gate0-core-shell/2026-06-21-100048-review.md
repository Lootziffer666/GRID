# Gate 0: Core shell and modular Split-APK restructuring

The GRID codebase is being restructured from a monolithic "Painkiller" Android app into a Split-APK architecture. This change renames the project, introduces a `:shared` pure-Kotlin module with a `ModuleRegistry`, `CrashBoundary`, and `ConfigLoader`, creates a `:feature:import` dynamic feature module containing all moved feature code (SAF readers, GitHub API clients, auth, upload flow), and strips `:app` down to a pure navigation shell. The `:domain` module keeps its existing `com.painkiller.domain` package to avoid churn.

**Watch for:** `TemplateRegistry.replaceAll` has a clear-then-repopulate window where concurrent readers see an empty registry (confirmed). `GridNavGraph` reads the module list directly from a `ConcurrentHashMap` with no Compose-observable state, so the UI won't recompose when modules register after initial composition (confirmed). No automatic feature module registration mechanism exists â€” `ImportFeatureEntry` is defined but never instantiated or registered anywhere at runtime (confirmed).

## High-level view

The module boundary is clean â€” no feature code (SAF, Ktor, GitHub, crypto) leaked into `:app/src/main`, verified by search. The dependency split in Gradle is consistent: `:app` retains only shell dependencies, `:feature:import` picks up everything the moved code needs, `:shared` has zero Android dependencies.

`ModuleRegistry` achieves per-operation thread safety via `ConcurrentHashMap`. `TemplateRegistry.replaceAll` does not â€” it uses `clear()` then iterates, creating a window where concurrent readers see an empty registry during hot-swap.

`ImportFeatureEntry` exists in `:feature:import` but nothing instantiates or registers it. The system is structurally correct but not end-to-end functional for feature invocation. The nav graph reads the registry once at composition time with no reactive bridge, so dynamically-registered modules won't appear without an external recomposition trigger.

`ConfigLoader` silently swallows missing resources and files, which may hide configuration bugs. `themes.xml` still references `painkiller_` resource names â€” functional but inconsistent with the rename.

Not tested: `ConfigLoader`, `TemplateRegistry`, malformed JSON handling.

## TemplateRegistry atomicity gap

`TemplateRegistry.replaceAll` calls `templates.clear()` then iterates the new list calling `templates[it.id] = it`. Between `clear()` and the last `put`, any concurrent call to `getById` or `listAll` will see a partial (or empty) registry. Since this is advertised as supporting "hot-swap at runtime," the window matters. A safer pattern:

```kotlin
fun replaceAll(newTemplates: List<WorkflowTemplate>) {
    val replacement = ConcurrentHashMap<String, WorkflowTemplate>()
    newTemplates.forEach { replacement[it.id] = it }
    templates = replacement  // atomic reference swap
}
```

This requires `templates` to be a `@Volatile var` rather than a `val`, or to use `AtomicReference`.

`ModuleRegistry` doesn't have this problem â€” `register` and `disable` are individual `ConcurrentHashMap` operations that are independently atomic, and there's no bulk-replace operation.

## Feature module discovery gap

`ImportFeatureEntry` is a concrete class in `:feature:import` but nothing instantiates it or calls `container.moduleRegistry.register(ImportFeatureEntry())`. In a dynamic feature module architecture, the typical pattern is either a `ContentProvider` that self-registers on class loading, a `SplitInstallManager` callback, or an explicit initializer in `GridApplication`. None of these exist yet. The feature module will be installed in the APK (via `dynamicFeatures += listOf(":feature:import")` which means install-time delivery by default with `<dist:fusing dist:include="true"/>`), but the entry point never connects to the registry.

## NavGraph observability

`GridNavGraph` calls `container.moduleRegistry.listActive()` inside the composable body without wrapping it in any state-observing mechanism. Nothing triggers recomposition when the registry changes. If a feature module registers after the activity's `setContent` block runs, the home screen remains stale.

For later gates where modules may register asynchronously (e.g., after SplitInstall completes), this will need a `StateFlow<List<FeatureEntry>>` exposed from the registry or a Compose `snapshotFlow` bridge.

## ConfigLoader failure silence

Both loading paths return without any signal on failure:

```kotlin
fun loadFromEmbedded(resourcePath: String) {
    val stream = this::class.java.classLoader?.getResourceAsStream(resourcePath)
        ?: return  // silent
}

fun loadFromExternal(path: String) {
    val file = File(path)
    if (!file.exists() || !file.isFile) return  // silent
}
```

If a template file is expected but missing (misconfigured asset path, missing resource in APK), the app silently operates with no templates loaded. A `Result<Unit>` return type or a logging call would surface misconfiguration during development.

## Test coverage gaps

The `:shared` module has 7 `ModuleRegistryTest` cases and 5 `CrashBoundaryTest` cases covering registration, crash isolation, module survival, and non-registered module handling.

Not tested: `ConfigLoader` (embedded loading, external loading, malformed JSON, missing resource behavior), `TemplateRegistry` (no direct tests for `getById`, `listAll`, `replaceAll` concurrency). The app-level `SmokeTest` validates Gradle dependency wiring between `:app`, `:domain`, and `:shared`.
